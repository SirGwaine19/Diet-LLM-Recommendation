# Diet LLM Recommendation System – Project Structure & File Roles

This document lists each directory as a section, with each file as a subheading. Under each file: ROLE (what the file is for) and BLOCKS (detailed purpose of each part so you can answer any question about the codebase).

================================================================================
## How the project was created
================================================================================
The repository was built as a **college major project**: an end-to-end web app for diet tracking and LLM-assisted meal parsing, with a documented phased plan (see BUILD_GUIDE.md). Creation followed this order:

1. **Repository layout** – Top-level folders `backend/`, `frontend/`, `database/`, and `docs/` were created to separate API, UI, DB instructions, and report assets. Git was initialized with `.gitignore` and `.gitattributes` (consistent line endings, especially for shell scripts).

2. **Database** – PostgreSQL is the primary store. `docker-compose.yml` at the repo root provides a local Postgres 16 instance; `database/SETUP.md` documents Docker vs manual install. Schema is applied only through **Alembic** migrations (`backend/alembic/`), not ad-hoc `create_all` in production flow.

3. **Backend (Phase 1 MVP)** – A Python virtual environment and `backend/requirements.txt` pin FastAPI, Uvicorn, SQLAlchemy 2, Alembic, psycopg2, Pydantic, JWT/passlib, and the OpenAI SDK. `app/core/` added settings (`config.py`), engine/session (`database.py`), bcrypt/JWT (`security.py`), and `get_current_user` (`dependencies.py`). SQLAlchemy models under `app/models/` match tables created in migration `001_initial_schema`; `002_add_sequence_defaults` fixes PostgreSQL `id` auto-increment. Pydantic schemas (`app/schemas/`) define the API contract. Services implement LLM meal parsing (`llm_service`), food matching and nutrient math (`nutrition_service`), orchestration (`meal_service`, `user_service`, `recommendation_service`). Routes are versioned under `app/api/v1/` and mounted in `main.py` with CORS for the SPA.

4. **Frontend** – **Vite + React 18 + TypeScript** scaffold (`npm create vite` style layout): `index.html` entry, `vite.config.ts` dev server on port 3000 with `/api` proxy to the backend. UI uses **React Router** for routes, **TanStack Query** for server state, **Zustand** (persisted) for the JWT, **react-hook-form** for forms, and **axios** via a shared `api.ts` client. Pages and components under `src/` implement login/register, dashboard (today’s macros, weekly chart, daily LLM summary), profile, log meal, and meal history.

5. **Data seeding** – `backend/scripts/seed_foods.py` loads a curated `COMMON_FOODS` list so parsed meal items can match reference foods and compute nutrients.

6. **Documentation and deliverables** – BUILD_GUIDE.md gives step-by-step replication; README.md is the short overview; PROJECT_STRUCTURE.txt (this file) is the file-by-file map; additional text/markdown files support reports, presentations, and methodology (see Root files and docs/).

## backend/
================================================================================
Backend root: Python FastAPI application. Run with `uvicorn app.main:app --reload` from backend/ with venv activated. Contains the app package, Alembic migrations, and utility scripts.

--------------------------------------------------------------------------------
### backend/requirements.txt
--------------------------------------------------------------------------------
ROLE: Pinned Python dependencies for reproducible installs (`pip install -r requirements.txt`). Lists FastAPI, Uvicorn, SQLAlchemy, Alembic, psycopg2-binary, python-dotenv, Pydantic, email-validator, python-jose, passlib/bcrypt, python-multipart, OpenAI client, httpx.

--------------------------------------------------------------------------------
### backend/alembic.ini
--------------------------------------------------------------------------------
ROLE: Alembic’s default config file (logging, script location). `sqlalchemy.url` is overridden at runtime by `alembic/env.py` from `settings.DATABASE_URL` so one `.env` drives both app and migrations.

--------------------------------------------------------------------------------
### backend/setup_venv.sh
--------------------------------------------------------------------------------
ROLE: Optional helper script (Unix/WSL) to create and activate a virtual environment in `backend/venv` and install requirements—convenience for developers who prefer a one-liner over manual `python -m venv`.

--------------------------------------------------------------------------------
### backend/app/
--------------------------------------------------------------------------------
Main application package: API routes, core (config, DB, security, dependencies), SQLAlchemy models, Pydantic schemas, and business-logic services.

--------------------------------------------------------------------------------
### backend/app/main.py
--------------------------------------------------------------------------------
ROLE: Entry point of the FastAPI application. Creates the app instance, configures CORS and routes; defines root and health endpoints.

BLOCKS:
- Imports: api_router from app.api.v1, settings from app.core.config, SessionLocal from app.core.database, FastAPI, CORSMiddleware, text from sqlalchemy.
- app = FastAPI(title="Diet LLM Recommendation System", description=..., version="1.0.0"): Creates the ASGI app; title/description appear in /docs (Swagger).
- app.add_middleware(CORSMiddleware, allow_origins=settings.CORS_ORIGINS, allow_credentials=True, allow_methods=["*"], allow_headers=["*"]): Lets the frontend (e.g. localhost:3000) call the API without browser blocking; credentials=true allows cookies/auth headers.
- app.include_router(api_router, prefix="/api/v1"): All API routes live under /api/v1 (e.g. /api/v1/auth/register, /api/v1/meals/log).
- root(): GET / – Returns JSON with message, version "1.0.0", status "running", docs link "/docs", health link "/health". No auth required.
- health_check(): GET /health – Sets db_status = "healthy", then opens SessionLocal(), runs db.execute(text("SELECT 1")), closes session; on any exception sets db_status = "unhealthy". Returns {"status": "healthy", "database": db_status}. Used for load balancers or monitoring; no auth.

--------------------------------------------------------------------------------
### backend/app/core/
--------------------------------------------------------------------------------
Core utilities used across the app: configuration from env, database engine/sessions, password hashing and JWT, and the get_current_user dependency for protected routes.

--------------------------------------------------------------------------------
### backend/app/core/config.py
--------------------------------------------------------------------------------
ROLE: Loads application settings from environment variables and the .env file. Used everywhere settings are needed (database, security, OpenAI, CORS).

BLOCKS:
- Settings(BaseSettings): Pydantic class that validates and loads env vars. Fields: DATABASE_URL (str, required), SECRET_KEY (str, required), ALGORITHM (str, default "HS256"), ACCESS_TOKEN_EXPIRE_MINUTES (int, default 30), OPENAI_API_KEY (str, required), DEBUG (bool, default True), CORS_ORIGINS (List[str], default ["http://localhost:3000", "http://localhost:8000"]). Missing required fields raise at startup.
- class Config: env_file = ".env" (relative to where the app is run, typically backend/), case_sensitive = True.
- settings = Settings(): Single instance; importing settings from app.core.config gives this object. DATABASE_URL is used by database.py and Alembic; SECRET_KEY and ALGORITHM by security.py; OPENAI_API_KEY by llm_service and recommendation_service; CORS_ORIGINS by main.py.

--------------------------------------------------------------------------------
### backend/app/core/database.py
--------------------------------------------------------------------------------
ROLE: SQLAlchemy engine, session factory, declarative base for models, and get_db dependency. All ORM models inherit from Base; routes use get_db to get a per-request session.

BLOCKS:
- create_engine(settings.DATABASE_URL, pool_pre_ping=True, echo=settings.DEBUG): Builds the PostgreSQL engine. pool_pre_ping=True tests connections before use (avoids stale connections). echo=True in DEBUG logs every SQL statement.
- SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine): Creates a session factory; each session is a single unit of work; autocommit/autoflush false so we control commits.
- Base = declarative_base(): All models (User, Meal, etc.) inherit from Base; Base.metadata is used by Alembic to generate migrations.
- get_db(): Generator that yields one session; typically used as Depends(get_db). After the request the finally block runs db.close(). If you don't use yield and close, connections can leak.

--------------------------------------------------------------------------------
### backend/app/core/security.py
--------------------------------------------------------------------------------
ROLE: Password hashing (bcrypt), JWT creation and verification, and safe handling of long passwords so bcrypt never receives more than 72 bytes.

BLOCKS:
- pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto"): Passlib context; "bcrypt" is the hashing algorithm; deprecated="auto" allows newer algorithms if configured.
- BCRYPT_MAX_PASSWORD_BYTES = 72: bcrypt algorithm only uses first 72 bytes; longer passwords would raise ValueError in some bcrypt versions without truncation.
- _truncate_for_bcrypt(password: str) -> str: Converts password to UTF-8 bytes; if len(encoded) > 72, truncates to 72 bytes and decodes with errors="replace" (avoids invalid UTF-8). Used so both hash and verify see the same truncated value.
- verify_password(plain_password, hashed_password) -> bool: Truncates plain_password, then calls pwd_context.verify(plain, hashed). Used in login to check password.
- get_password_hash(password) -> str: Truncates password, then pwd_context.hash(...). Used in register and anywhere we store a new password.
- create_access_token(subject, expires_delta=None) -> str: subject is typically user.id (stringified). If expires_delta not given, uses settings.ACCESS_TOKEN_EXPIRE_MINUTES. Builds payload {"exp": expire, "sub": str(subject)}; returns jwt.encode(payload, settings.SECRET_KEY, algorithm=settings.ALGORITHM). Used after register and login.
- verify_token(token) -> Optional[str]: jwt.decode with SECRET_KEY and ALGORITHM; returns payload.get("sub") (user id string) or None on JWTError (invalid/expired). Used by get_current_user.

--------------------------------------------------------------------------------
### backend/app/core/dependencies.py
--------------------------------------------------------------------------------
ROLE: FastAPI dependency get_current_user: extracts Bearer token from the request, verifies it, loads the User from DB, and returns it so route handlers can use the authenticated user.

BLOCKS:
- security = HTTPBearer(): FastAPI security scheme; automatically returns 403 if Authorization header is missing or not Bearer; injects HTTPAuthorizationCredentials into the dependency.
- get_current_user(credentials: HTTPAuthorizationCredentials, db: Session) -> User: (1) token = credentials.credentials. (2) subject = verify_token(token); if None, raise HTTPException 401 "Invalid or expired token" with WWW-Authenticate: Bearer. (3) user = db.query(User).filter(User.id == int(subject)).first(); if None, raise 401 "User not found". (4) If not user.is_active, raise 403 "Inactive user". (5) Return user. Used by all routes under users, meals, recommendations (via router-level dependencies in api/v1/__init__.py).

### backend/app/models/
--------------------------------------------------------------------------------
SQLAlchemy ORM models. Each class maps to one PostgreSQL table; columns define schema; relationships define links between tables. Alembic uses Base.metadata to generate migrations; the app never creates tables at runtime (migrations do).

--------------------------------------------------------------------------------
### backend/app/models/__init__.py
--------------------------------------------------------------------------------
ROLE: Re-exports all models so other code can do "from app.models import User, Meal, ...". Alembic env.py imports from here to attach target_metadata.

BLOCKS:
- Imports: User from app.models.user, Meal from app.models.meal, MealItem from app.models.meal_item, NutrientAggregate from app.models.nutrient, Recommendation from app.models.recommendation, Food from app.models.food.
- __all__ = ["User", "Meal", "MealItem", "NutrientAggregate", "Recommendation", "Food"]: Public API of the package.

--------------------------------------------------------------------------------
### backend/app/models/user.py
--------------------------------------------------------------------------------
ROLE: User account and profile (table: users). Stores credentials, profile fields, dietary prefs, and daily goals. Referenced by meals and recommendations.

BLOCKS:
- class User(Base), __tablename__ = "users".
- id: Integer, primary_key, index. Filled by DB sequence (migration 002).
- email: String, unique=True, index=True, nullable=False. Used for login lookup.
- hashed_password: String, nullable=False. bcrypt hash from security.get_password_hash.
- full_name: String, nullable.
- Profile: age (Integer), sex (String), height_cm (Float), weight_kg (Float), target_weight_kg (Float), activity_level (String), all nullable.
- dietary_preferences, allergies, cultural_preferences: JSON columns, default=list. Stored as Python lists (e.g. ["vegetarian", "nuts"]).
- daily_calorie_target (Integer), protein_target_g, carb_target_g, fat_target_g (Float), nullable. Used for dashboard progress and recommendation prompts.
- is_active: Boolean, default True. If False, login returns 403.
- created_at, updated_at: DateTime(timezone=True), server_default=func.now() and onupdate=func.now().
- meals: relationship("Meal", back_populates="user", cascade="all, delete-orphan"). One user many meals; deleting user deletes meals.
- recommendations: relationship("Recommendation", back_populates="user", cascade="all, delete-orphan").

--------------------------------------------------------------------------------
### backend/app/models/meal.py
--------------------------------------------------------------------------------
ROLE: One logged meal (table: meals). Belongs to a user; has many meal_items and one nutrient_aggregate.

BLOCKS:
- id, user_id (FK users.id, nullable=False), timestamp (server_default=func.now()), meal_type (e.g. "breakfast"), source (default "text" for text logging).
- user: relationship("User", back_populates="meals").
- meal_items: relationship("MealItem", back_populates="meal", cascade="all, delete-orphan").
- nutrient_aggregate: relationship("NutrientAggregate", back_populates="meal", uselist=False, cascade="all, delete-orphan"). uselist=False means one-to-one.

--------------------------------------------------------------------------------
### backend/app/models/meal_item.py
--------------------------------------------------------------------------------
ROLE: Single food item in a meal (table: meal_items). Parsed from LLM; may link to foods table via normalized_food_id for nutrition lookup.

BLOCKS:
- id, meal_id (FK meals.id), food_name (string from LLM), normalized_food_id (FK foods.id, nullable – set when match_food_item finds a match), quantity (float, default 1.0), unit (optional string), portion_size_category, preparation_method, confidence_score (optional float from LLM).
- meal: relationship("Meal", back_populates="meal_items").

--------------------------------------------------------------------------------
### backend/app/models/nutrient.py
--------------------------------------------------------------------------------
ROLE: Aggregated nutrients for one meal (table: nutrient_aggregates). One row per meal; stores total calories, protein_g, carbs_g, fat_g, fiber_g, sodium_mg for that meal. Used for daily totals and display.

BLOCKS:
- id, meal_id (FK meals, nullable), user_id (FK users), date (Date). date is the calendar day of the meal for aggregation.
- calories, protein_g, carbs_g, fat_g, fiber_g, sodium_mg: Float, nullable, default 0.
- created_at: DateTime(timezone=True), server_default=func.now().
- meal: relationship("Meal", back_populates="nutrient_aggregate").

--------------------------------------------------------------------------------
### backend/app/models/recommendation.py
--------------------------------------------------------------------------------
ROLE: One generated recommendation/summary (table: recommendations). type "daily" for daily summary; content is the LLM text; user_feedback stores "liked"/"ignored" etc.

BLOCKS:
- id, user_id (FK users), type (e.g. "daily"), content (Text), generated_at (server_default=func.now()), user_feedback (String, nullable).
- extra_metadata: Column("metadata", JSON, nullable=True). Python attribute is extra_metadata because "metadata" is reserved by SQLAlchemy; DB column name is "metadata". Stores e.g. {"date": "...", "stats": {...}}.
- user: relationship("User", back_populates="recommendations").

--------------------------------------------------------------------------------
### backend/app/models/food.py
--------------------------------------------------------------------------------
ROLE: Reference food for nutrition (table: foods). Seeded by scripts/seed_foods.py; nutrition_service matches meal item names to Food and uses per-100g values to compute nutrients.

BLOCKS:
- id, name (String, index), description (optional).
- calories_per_100g, protein_per_100g, carbs_per_100g, fat_per_100g, fiber_per_100g, sodium_per_100g_mg: Float, nullable, default 0. All nutrients are per 100g; quantity and unit in meal_item determine the factor in nutrition_service.

--------------------------------------------------------------------------------
Pydantic models for request body validation and response serialization. They define the API contract (what the client sends and what the server returns). No database access; used in route signatures and response_model.

### backend/app/schemas/auth.py

--------------------------------------------------------------------------------
### backend/app/schemas/user.py
--------------------------------------------------------------------------------
ROLE: User profile and goals: request (update) and response (full user) schemas.

BLOCKS:
- UserBase: email, full_name, age, sex, height_cm, weight_kg, target_weight_kg, activity_level, dietary_preferences (List[str]), allergies (List[str]), cultural_preferences (List[str]). Base for create/response.

- UserGoalsUpdate: daily_calorie_target, protein_target_g, carb_target_g, fat_target_g all optional. For PUT /users/me/goals.
- UserResponse: id, email, full_name, age, sex, height_cm, weight_kg, target_weight_kg, activity_level, dietary_preferences, allergies, cultural_preferences, daily_calorie_target, protein_target_g, carb_target_g, fat_target_g. Config from_attributes=True so FastAPI can build from ORM User.

--------------------------------------------------------------------------------
### backend/app/schemas/meal.py
--------------------------------------------------------------------------------
ROLE: Meal log request (text + optional meal_type) and meal/item response with nutrients.

BLOCKS:
- MealItemCreate: food_name, quantity (default 1.0), unit, portion_size_category, preparation_method, confidence_score. Matches LLM output structure; used internally in meal_service when building MealItems.

- MealResponse: id, user_id, timestamp, meal_type, source, items (list), calories, protein_g, carbs_g, fat_g (all optional for nutrients). response_model for log, get_meals, get_meal.

### backend/app/schemas/recommendation.py

ROLE: Recommendation response and feedback request.

BLOCKS:

### backend/app/api/
--------------------------------------------------------------------------------
API route modules. v1 is the current version; all routes are under /api/v1.

--------------------------------------------------------------------------------
### backend/app/api/v1/__init__.py
--------------------------------------------------------------------------------
ROLE: Builds the single api_router that main.py mounts at /api/v1. Imports auth, users, meals, recommendations and applies prefixes and optional auth dependency.

BLOCKS:
- api_router = APIRouter().
- include_router(auth.router, prefix="/auth", tags=["auth"]): No dependencies list, so register and login are public; GET /auth/me uses get_current_user on the route itself in auth.py.
- include_router(users.router, prefix="/users", tags=["users"], dependencies=[Depends(get_current_user)]): Every route in users (get_me, update_me, get_goals, put_goals) requires a valid JWT.

- include_router(recommendations.router, prefix="/recommendations", tags=["recommendations"], dependencies=[Depends(get_current_user)]). Resulting paths: /api/v1/auth/..., /api/v1/users/..., /api/v1/meals/..., /api/v1/recommendations/....

--------------------------------------------------------------------------------
### backend/app/api/v1/auth.py
--------------------------------------------------------------------------------
ROLE: User registration, login, and current-user info. Register and login return a JWT; /me returns minimal user info and requires auth.

BLOCKS:
- register(data: UserRegister, db: Session = Depends(get_db)): (1) If db.query(User).filter(User.email == data.email).first() exists, raise HTTPException 400 "Email already registered". (2) user = User(email=data.email, hashed_password=get_password_hash(data.password), full_name=data.full_name). (3) db.add(user), db.commit(), db.refresh(user). (4) token = create_access_token(subject=str(user.id)). (5) return Token(access_token=token). Response model Token so only access_token and token_type are returned.
- login(data: UserLogin, db): (1) user = db.query(User).filter(User.email == data.email).first(). (2) If not user or not verify_password(data.password, user.hashed_password), raise 401 "Invalid email or password". (3) If not user.is_active raise 403 "Inactive user". (4) return Token(access_token=create_access_token(subject=str(user.id))).
- me(current_user: User = Depends(get_current_user)): Returns dict with id, email, full_name. No DB query; user comes from dependency.

--------------------------------------------------------------------------------
### backend/app/api/v1/users.py
--------------------------------------------------------------------------------
ROLE: Get/update current user profile and get/update goals. All routes require authentication (router-level dependency).

BLOCKS:
- get_me(current_user, ...): Returns current_user; FastAPI serializes to UserResponse (all profile and goal fields).
- update_me(data: UserUpdate, current_user, db): Calls update_user_profile(db, current_user, data); returns updated UserResponse. Only fields sent in body are updated.
- get_goals(current_user): Returns dict with daily_calorie_target, protein_target_g, carb_target_g, fat_target_g from current_user.
- put_goals(data: UserGoalsUpdate, current_user, db): Calls update_user_goals(db, current_user, data); returns the same shape as get_goals.

--------------------------------------------------------------------------------
### backend/app/api/v1/meals.py
--------------------------------------------------------------------------------
ROLE: Log a meal from text (via LLM + nutrition DB), list meals, get one meal, delete meal. All routes require auth. Errors from meal logging are caught and turned into user-friendly HTTP errors.

BLOCKS:
- log_meal(data: MealLogRequest, current_user, db): (1) try: meal = log_meal_from_text(db, current_user, data.text, data.meal_type). (2) except: logger.exception(...); err_msg = str(e). If "insufficient_quota" or "429" in err_msg raise 503 "OpenAI quota exceeded. Add billing at ...". If "openai" or "api_key" or "authentication" in err_msg raise 503 "Meal parsing failed: check OPENAI_API_KEY...". Else raise 500 with detail="Meal logging failed: {err_msg}". (3) On success: build MealResponse from meal (id, user_id, timestamp, meal_type, source), items from meal.meal_items (id, food_name, quantity, unit, portion_size_category, preparation_method), and calories/protein_g/carbs_g/fat_g from meal.nutrient_aggregate. Return MealResponse.
- get_meals(current_user, db, limit=50, start_date=None, end_date=None): Base query filters by user_id; optional start_date/end_date filter `func.date(Meal.timestamp)` inclusive for weekly or ranged lists. Order by timestamp desc, limit. Map each to MealResponse (same shape as log_meal). Used by the dashboard for “today” and “week” windows.
- get_meal(meal_id, current_user, db): Query Meal with id=meal_id and user_id=current_user.id. If not found raise 404 "Meal not found". Else return MealResponse.
- delete_meal(meal_id, current_user, db): Same query; if not found 404; else db.delete(meal), db.commit(), return {"ok": True}. Cascade deletes meal_items and nutrient_aggregate.

--------------------------------------------------------------------------------
### backend/app/api/v1/recommendations.py
--------------------------------------------------------------------------------
ROLE: Get today's daily summary (if any), generate a new one (LLM), and submit feedback (liked/ignored). All require auth.

BLOCKS:
- get_daily(current_user, db): Query Recommendation where user_id=current_user.id, type="daily", order_by(Recommendation.generated_at.desc()), limit 1. Return first or None (FastAPI returns null in JSON).
- generate(current_user, db): rec = generate_daily_summary(db, current_user, date.today()); return rec (RecommendationResponse). This calls OpenAI and creates one new Recommendation row.
- feedback(recommendation_id, data: RecommendationFeedback, current_user, db): Query Recommendation by id and user_id; if not found 404 "Not found"; else set rec.user_feedback = data.feedback, db.commit(), return {"ok": True}.

--------------------------------------------------------------------------------
### backend/app/services/
--------------------------------------------------------------------------------
Business logic: no HTTP or request context; they receive db session and domain objects. Called by API route handlers.

--------------------------------------------------------------------------------
### backend/app/services/llm_service.py
--------------------------------------------------------------------------------
ROLE: Parse free-form meal description (e.g. "2 eggs and toast") into structured JSON using OpenAI. Used only by meal_service when logging a meal.

BLOCKS:
- client = OpenAI(api_key=settings.OPENAI_API_KEY): Single client instance. Fails at import if OPENAI_API_KEY missing.
- MEAL_PARSE_PROMPT: String instructions for the LLM. Tells it to extract food items with food_name, quantity, unit, portion_size_category, preparation_method, confidence_score; meal_type if inferrable; output valid JSON only; never fabricate nutrients. Example JSON shape shown in the prompt.
- parse_meal_text(text: str) -> dict: (1) response = client.chat.completions.create(model="gpt-4o-mini", messages=[system "You extract...", user MEAL_PARSE_PROMPT + text], temperature=0.2). (2) content = response.choices[0].message.content.strip(). (3) If content starts with ``` (markdown), strip the first code fence and optional "json" label. (4) return json.loads(content). Raises if OpenAI errors (e.g. 429 quota) or JSON invalid. Returned dict has "meal_type" (optional) and "items" (list of dicts with food_name, quantity, unit, etc.).

--------------------------------------------------------------------------------
### backend/app/services/nutrition_service.py
--------------------------------------------------------------------------------
ROLE: Match a food name (from LLM) to the foods table and compute nutrients for items and for a full meal. Used by meal_service after parsing and when building NutrientAggregate.

BLOCKS:
- search_food(db, query, limit=5): q = "%{query.lower()}%"; return db.query(Food).filter(Food.name.ilike(q)).limit(limit).all(). Used for fuzzy name match.
- match_food_item(db, food_name): (1) Try search_food(db, name). If results, return results[0]. (2) Else try first word of name (e.g. "scrambled eggs" -> "eggs"). (3) Else try singular (strip trailing 's') or plural (add 's') and search again. Return first match or None.
- _SERVING_UNITS: Tuple of units that mean "number of servings": cup, cups, bowl, bowls, serving, servings, portion, portions, slice, slices, piece, pieces, item, items, pizza, pizzas. Used in calculate_nutrients_for_item.
- calculate_nutrients_for_item(food, quantity, unit): unit_lower = (unit or "").strip().lower(). If unit in ("g", "gram", "grams", "gr"): factor = quantity / 100.0 (treat quantity as grams). Elif unit in _SERVING_UNITS or unit empty: factor = max(0.1, quantity) (quantity = number of 100g servings, e.g. "2 pizza" -> factor 2). Else: factor = max(0.1, quantity). Return dict with calories, protein_g, carbs_g, fat_g, fiber_g, sodium_mg each = (food.*_per_100g or 0) * factor.
- _DEFAULT_PER_SERVING: Dict with default calories 80, protein_g 4, carbs_g 10, fat_g 2, fiber_g 1, sodium_mg 50. Used when an item has no matching Food so totals are not zero.
- calculate_meal_nutrients(db, meal_items): total = {calories: 0, protein_g: 0, ...}. For each item: if item.normalized_food_id, load Food and add calculate_nutrients_for_item(food, item.quantity, item.unit) to total; else add _DEFAULT_PER_SERVING * max(0.1, item.quantity) to total.
