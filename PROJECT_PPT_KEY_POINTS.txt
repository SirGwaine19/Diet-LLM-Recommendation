================================================================================
DIET LLM RECOMMENDATION SYSTEM - KEY POINTS
================================================================================

0. STATEMENT OF PURPOSE AND OBJECTIVES
--------------------------------------------------------------------------------

• Purpose and objectives
  → To build an LLM-powered diet recommendation system that allows users to log
    meals in natural language (text or image), obtain accurate nutrition data
    from a USDA-backed database, and receive personalized, actionable dietary
    feedback. Objectives: (1) reduce logging friction, (2) improve nutrient
    accuracy via structured extraction + matching, (3) deliver explainable,
    goal-aligned recommendations.

• Problem and relevance
  → Manual calorie logging is tedious; existing apps use rigid input (fixed
    menus, barcode scans) that don't match how people describe meals. Users
    abandon logging quickly, and generic advice ignores goals, allergies, and
    preferences. This project addresses real-world diet adherence and
    personalization gaps using modern NLP and nutrition data.

• Scope boundaries
  IN SCOPE:  Text and image meal logging, LLM extraction, USDA/food DB
    matching, daily nutrient summaries, RAG-based coaching, user profiles
    (goals, allergies), web prototype (FastAPI + React).
  OUT OF SCOPE:  Medical diagnosis/treatment, prescription diets, real-time
    barcode lookup, offline mobile app, meal planning/grocery automation
    (future extension only).

• Key deliverables
  → Working web application (prototype)
  → Nutrition database (USDA + curated entries)
  → LLM extraction + matching pipeline
  → REST API and authenticated frontend
  → Project report / documentation

• Target users and beneficiaries
  → Health-conscious individuals tracking diet
  → Users with dietary constraints (vegetarian, keto, allergies)
  → Anyone seeking low-friction, natural-language meal logging
  → People who want personalized, non-judgmental nutrition feedback

• Expected outcomes and success criteria
  → Time-to-log under 30 seconds per meal (vs 2–3 min in traditional apps)
  → Accurate macro estimates (validated against USDA ground truth)
  → User retention and adherence improvement over baseline
  → Modular, scalable architecture suitable for future features

• Technologies and tools
  → Backend: FastAPI, PostgreSQL, SQLAlchemy, Alembic, JWT
  → AI: LLM (OpenAI), embeddings, RAG over nutrition KB
  → Frontend: React, modern web stack
  → Deployment: Docker, docker-compose

• Assumptions and constraints
  → Assumptions: Users have internet access; USDA data sufficient for common
    foods; LLM outputs can be constrained to structured JSON; users provide
    reasonably descriptive meal text.
  → Constraints: No medical advice; system is advisory only; API rate limits
    and cost constraints for LLM calls; nutrition DB coverage limits accuracy
    for obscure/regional foods.

--------------------------------------------------------------------------------

1. WHY THIS WORK
• Manual logging tedious; apps use rigid input
  → Most diet apps require selecting from fixed lists, barcode scans, or manual
    calorie entry. This friction causes users to abandon logging after a few days.

• Users want natural language ("2 slices pizza, Coke")
  → People think and speak in phrases like "had a couple slices and a Coke."
    Apps should accept this directly instead of forcing structured menus.

• Need personalized, actionable feedback from AI
  → Generic calorie targets are not enough. Users need suggestions that fit
    their lifestyle, goals, and preferences—e.g., "Try adding a side salad."

• LLM + nutrition DB for accurate, explainable recommendations
  → LLMs handle natural language; nutrition DBs provide reliable numbers.
    Combining both gives both accuracy and human-friendly explanations.

• Existing apps lack contextual understanding of meals & portions
  → Apps struggle with "medium plate," "restaurant portion," or "homemade."
    They can't infer portion sizes or regional/cultural food variations.

• Diet adherence drops when logging feels like a chore
  → Research shows that friction in logging is a major reason people quit.
    Lower friction = higher long-term engagement and better outcomes.

• One-size-fits-all advice doesn't account for individual preferences
  → Vegans, keto dieters, and people with allergies need tailored advice.
    Generic recommendations are less useful and can even be unsafe.

2. WHAT YOU PLAN TO DO
• LLM-powered diet recommendation system
  → Build an end-to-end system where an LLM parses meals, matches to a DB,
    computes nutrients, and generates personalized diet feedback.

• Text + image meal logging (natural language / photo)
  → Users log via free-form text ("chicken biryani, raita") or by uploading
    a photo. The LLM extracts food items and estimates portions from both.

• Map foods → nutrition DB (USDA) → daily summaries
  → Each extracted food is matched to USDA (or similar) entries for accurate
    calories, macros, and micronutrients. Daily totals and summaries are shown.

• RAG-based coaching; user profiles (goals, allergies)
  → Use retrieval-augmented generation over a nutrition knowledge base to give
    coaching tips. User profiles store goals, allergies, and constraints.

• Interpret vague inputs ("home-cooked curry", "medium bowl")
  → The system uses LLM reasoning + semantic search to infer likely foods and
    portion sizes even when descriptions are imprecise.

• Weekly/monthly nutrition trends and insights
  → Beyond daily totals: show trends (e.g., protein intake over time),
    gaps (e.g., low fiber), and insights to guide behavior change.

• Dietary constraint handling (vegan, keto, diabetic-friendly)
  → Recommendations and summaries respect user preferences: e.g., flag
    non-vegan items, suggest keto swaps, or highlight carbs for diabetics.

3. HOW - APPROACH
• LLM structured extraction → fuzzy/semantic match → nutrient calc
  → LLM extracts structured food items (name, quantity, unit) from text/image.
    These are matched to the DB via fuzzy/semantic search; nutrients are summed.

• RAG over nutrition KB
  → A nutrition knowledge base is indexed; RAG retrieves relevant facts to
    ground LLM responses and keep advice consistent and evidence-based.

• FastAPI, PostgreSQL, JWT
  → FastAPI for REST APIs, PostgreSQL for users, logs, and food DB, JWT for
    secure, stateless authentication.

• Safe prompting (non-judgmental, no medical advice)
  → Prompts instruct the LLM to be supportive, avoid moralizing, and clearly
    state that the system is not a substitute for medical advice.

• Semantic search for similar foods when exact match missing
  → If "chicken tikka masala" is not in the DB, find similar items (e.g.,
    "chicken curry," "grilled chicken") and estimate nutrients.

• Explainable outputs (cite sources, show reasoning)
  → Recommendations cite DB entries or KB snippets. Users see why a food was
    matched and how nutrients were calculated.

• Modular pipeline: extraction → matching → aggregation → feedback
  → Each step (parse, match, aggregate, generate feedback) is separate.
    Easier to debug, improve, and swap components (e.g., different LLMs).

4. EXPECTED RESULTS
• Low-friction logging
  → Users spend seconds, not minutes, logging a meal. Natural language and
    photos remove the need for rigid menus and manual calorie entry.

• Accurate nutrients, personalized summaries
  → USDA-backed data ensures reliable calories and macros. Summaries are
    tailored to each user's goals and constraints.

• Better adherence; scalable architecture
  → Lower friction and personalized feedback improve long-term use. The
    modular design supports scaling to more users and features.

• Reduced time-to-log vs traditional apps
  → Target: log a typical meal in under 30 seconds vs. 2–3 minutes with
    traditional calorie-tracking apps.

• Transparent recommendations users can trust
  → Because we cite sources and show reasoning, users understand where
    numbers and advice come from—building trust and acceptance.

• Extensible design for new features (recipes, grocery lists)
  → The same pipeline can power meal planning, recipe suggestions, or
    grocery lists without major rework.

• Modules and data flow
  → CLIENT: User logs meal (text/photo) → sends to API → displays nutrients and
    recommendations. Data: MealLogRequest, MealResponse, RecommendationResponse.
  → API: Validates JWT, routes to services, returns JSON. Data: HTTP requests/responses.
  → AUTH MODULE: Registers user, issues JWT, validates token on each request.
  → USER SERVICE: CRUD profile, goals, preferences. Reads/writes users table.
  → MEAL SERVICE: Orchestrates LLM extraction → nutrition matching → storage.
    Data flow: text → LLM → MealCreate (items) → NutritionService.match → MealItem
    → NutrientAggregate → Meal persisted.
  → NUTRITION SERVICE: Fuzzy/semantic match food name → Food; compute nutrients
    per item (factor × per_100g) → aggregate. Reads foods table.
  → RECOMMENDATION SERVICE: Aggregates daily nutrients, builds prompt with user
    profile + RAG context → LLM → Recommendation stored.
  → DATA LAYER: PostgreSQL stores all entities; optional vector store for RAG.

7. CORE LOGIC / ALGORITHM (4–8 Step Workflow and Flowchart)
--------------------------------------------------------------------------------

• Meal logging pipeline (6 steps)
  Step 1: RECEIVE INPUT — User submits meal text (or image URL) + optional meal_type.
  Step 2: EXTRACT — LLM parses text → structured JSON: [{food_name, quantity, unit,
          portion_size_category, preparation_method, confidence_score}, ...]
  Step 3: MATCH — For each item: fuzzy/semantic search over foods table → best
          Food entry (normalized_food_id). Fallback: estimate from similar food.
  Step 4: CALCULATE — factor = quantity/100 (grams) or quantity×serving_factor;
          nutrients = Food.per_100g × factor; sum across items → meal totals.
  Step 5: STORE — Create Meal, MealItems (with normalized_food_id), NutrientAggregate.
  Step 6: RESPOND — Return MealResponse with items + nutrients.

  Step 1: Aggregate daily nutrients for user + date.

  Step 4: LLM generates summary → store Recommendation → return to user.

9. IMPLEMENTATION STRATEGY
--------------------------------------------------------------------------------

• FastAPI — async, OpenAPI docs, Pydantic validation, DI
  → High performance, auto-generated docs, easy validation, and clean dependency injection for DB/auth.

• PostgreSQL — ACID, JSON, relational schema, pgvector-ready
  → Reliable storage, JSON columns for preferences, and future support for vector embeddings.

• JWT + bcrypt — stateless auth, scalable
  → No server-side sessions; works across multiple instances; industry-standard password hashing.

• Modular services — single responsibility, swappable, testable
  → Each service does one job; easy to mock in tests and swap components (e.g. different LLM).

• Architecture — Client → API → services → DB (clear separation)
  → Frontend calls API; API delegates to services; services use DB—easy to extend and maintain.

10. CODE EVIDENCE
--------------------------------------------------------------------------------

• LLM extraction: parse_meal_text(text) → JSON with meal_type, items; strip
  markdown; extract food_name, quantity, unit, confidence_score.

• Food match + nutrients: full name → first word → singular/plural; factor
  = qty/100 (grams) or qty (servings); nutrients = per_100g × factor.

• Meal orchestration: parse → Meal + MealItems → match_food_item → aggregate
  nutrients → NutrientAggregate → commit.

11. WORKING FLOW (Input → Process → Output)
--------------------------------------------------------------------------------

• Meal logging flow
  Input:  Free-form text ("2 eggs and toast") + optional meal_type
  Process: parse_meal_text → match_food_item → calculate_nutrients → persist
  Output: MealResponse with items, calories, protein_g, carbs_g, fat_g

• Daily summary flow
  Input:  User + date (default: today)
  Process: aggregate daily nutrients → build prompt (profile, stats, history) → LLM
  Output: Recommendation (daily summary text, metadata with stats)

• Auth flow
  Input:  email, password
  Process: verify credentials → create_access_token
  Output: {access_token, token_type}

• Screenshot / screen recording
  → Capture: LogMeal page (text input + parsed result + nutrients), Dashboard
    (daily progress + summary), MealHistory (list), Profile (goals). Record
    end-to-end flow: login → log meal → view dashboard.
