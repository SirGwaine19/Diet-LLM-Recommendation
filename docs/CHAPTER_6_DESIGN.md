# CHAPTER 6: SYSTEM DESIGN AND IMPLEMENTATION

## 6.1 High Level Design

The proposed Diet LLM Recommendation System follows a modular high-level architecture that brings together multiple interconnected components in a unified intelligent platform. At the front end, a clean and responsive user interface enables interaction through natural language input, allowing users to describe their meals, ask dietary questions, or share information in a conversational manner without learning any technical syntax. The user input is then forwarded to a natural language processing layer powered by a Large Language Model, which interprets the query, extracts intent, and understands contextual signals such as time of day, meal type, and food identity. To maintain coherent multi-turn interactions, a dialogue management component keeps track of conversational state and ensures that each subsequent user message is interpreted with awareness of earlier exchanges.

Once the input has been understood, the processed query is forwarded to the nutrition analysis module, which integrates with structured food composition databases to retrieve accurate nutrient information for each identified food item. A personalization engine then refines the analysis by incorporating user-specific factors such as age, weight, height, activity level, dietary preferences, allergies, and health goals, ensuring that generated insights remain individually relevant. The recommendation engine builds upon this enriched analysis to produce meaningful outputs such as meal suggestions, calorie guidance, and dietary advice. To prepare the architecture for advanced future scenarios, optional multimodal capabilities are included in the design, where image input can be processed using computer vision models to identify food items directly from photographs.

The backend infrastructure, typically deployed in a cloud environment, handles structured data storage, model computation, secure communication, and external API integrations. Dedicated security and privacy modules ensure safe handling of personal information and dietary data, while an optimization layer focuses on reducing computational cost and improving overall response time. Finally, a feedback and evaluation module collects user responses to continuously refine system behavior and improve recommendation quality over time. Together, these components form a layered architecture that delivers scalability, efficiency, and adaptability suitable for real-world deployment.

A useful conceptual abstraction within this design is the cooperation of three intelligent roles operating across the pipeline. A Nutrition Analyst role hypothesizes meal composition from visuals or text, a Verifier role refines those hypotheses through database lookups and validation, and an Advisor role personalizes the final recommendation across multiple turns by referencing earlier dietary context, for example reasoning that a previously logged chicken meal already contributes a known calorie share to the daily goal. This iterative refinement strategy gracefully handles ambiguity, allowing the system to ask clarifying questions about portion sizes or unclear food terms instead of producing inaccurate outputs.

## 6.2 End-to-End User Workflow

The end-to-end user workflow has been designed to ensure that the system delivers a smooth, supportive, and continuously improving dietary guidance experience. The workflow begins when the user enters the system through the application interface, which serves as the primary point of interaction. Upon arrival, new users are guided through a registration process that creates a personal profile in the system, allowing all subsequent dietary records, recommendations, and analytics to be associated with that specific user.

After registration, the user is prompted to provide essential personal and health-related information, including age, weight, height, activity level, and dietary preferences. Using these inputs, the system calculates the user’s body mass index and estimates the daily calorie requirements that are appropriate for their physical profile and activity level. Building further on this foundation, the system performs a deeper nutritional requirement analysis, determining the desirable distribution of proteins, carbohydrates, fats, fiber, and supporting nutrients required to maintain balanced nutrition for the individual.

With these requirements in place, the system generates a personalized diet plan that reflects the user’s unique characteristics, lifestyle preferences, and health goals. This plan is then presented through the recommendations interface, where the user can clearly see meal suggestions, dietary advice, and intake targets in a structured and easy-to-understand format. Once the user begins logging meals and interacting with the recommendations, the system continuously gathers user feedback and monitors progress over time. This feedback loop is essential because it transforms the workflow from a one-time setup into a continuous improvement cycle, where each new meal, every user response, and every observed pattern is used to refine future recommendations. Although the formal flow ends when the user closes the session, the underlying personalization cycle keeps evolving across sessions, ensuring that the system grows progressively smarter and more aligned with the user over time.

## 6.3 LLM-Based Meal Extraction Algorithm

The LLM-based meal extraction algorithm is the primary intelligence engine of the Diet LLM Recommendation System. It is responsible for transforming free-form natural language meal descriptions written by the user into structured, machine-processable data that downstream modules can use for nutrient calculation, storage, and recommendation generation. This algorithm replaces the rigid manual food-search workflow of conventional diet trackers with a conversational, context-aware understanding approach driven by Large Language Models.

**Input:** A user-provided meal description \( T \) in natural language, such as “two rotis with dal and a glass of milk.”  
**Output:** A structured set of meal items \( M = \{(f_i, q_i, u_i, p_i, c_i)\}_{i=1}^{N} \), where for each detected item, \( f_i \) represents the food name, \( q_i \) the quantity, \( u_i \) the unit, \( p_i \) the portion or preparation indicator, and \( c_i \) the confidence score.

The algorithm begins with a preprocessing stage in which the input text is cleaned, normalized, and validated. Whitespace is trimmed, redundant punctuation is removed, and basic structural validation ensures that the input is non-empty and within acceptable length limits. The cleaned text is then injected into a carefully engineered system prompt that instructs the GPT-based Large Language Model to extract individual food items along with their quantities, units, portions, and a confidence score for each detection. The prompt is designed to enforce a strict JSON-style output schema, ensuring that downstream modules always receive consistent and predictable data.

In the next stage, the LLM produces structured output \( M \) representing the parsed meal. A confidence-based filtering rule is then applied, where any item with confidence \( c_i < \tau_{conf} \) is flagged for user confirmation rather than being silently accepted, ensuring that ambiguous detections do not corrupt downstream nutrient calculations. The filtered structured output is then passed to the food matching algorithm, while the original input is also retained for traceability and debugging. This design ensures that the meal extraction algorithm remains highly accurate, conservative under uncertainty, and seamlessly compatible with the rest of the pipeline.

## 6.4 Food Matching Algorithm

After the LLM produces structured meal items, the food matching algorithm is responsible for mapping each extracted food name to the corresponding entry in the curated nutrition database. This step is critical, because variations in spelling, regional naming, plural forms, and informal vocabulary can otherwise cause valid foods to remain unrecognized.

**Input:** A parsed food name \( f_i \) from the LLM output.  
**Output:** The most relevant nutrition database entry \( D_j \) and an associated match confidence score.

The algorithm performs the matching through a layered strategy. In the first stage, a fuzzy string match is performed using case-insensitive substring search and lightweight similarity scoring against the foods table, which can be approximated through a similarity function \( S_{fuzzy}(f_i, D_j) \) that returns a score between 0 and 1. If the highest fuzzy score exceeds a defined threshold \( \tau_{fuzzy} \), the corresponding food entry is selected as the match.

If fuzzy matching does not produce a confident result, the algorithm proceeds to a word-reduction stage, where the food name is simplified, for example by trying the first significant word only (so that “scrambled eggs” reduces to “eggs”). If this still does not produce a match, the algorithm tries singular and plural variants of the food name, allowing it to recognize foods regardless of grammatical form. Finally, if all of the above strategies fail to produce a confidence above a stricter threshold \( \tau_{accept} \), the item is flagged for user confirmation rather than being incorrectly mapped. This conservative design protects nutrient accuracy and provides a clear path toward future enhancement using semantic embedding-based matching introduced in Phase 2.

## 6.5 Nutrient Calculation Algorithm

The nutrient calculation algorithm computes the actual nutritional impact of every meal once its food items have been parsed and matched. This algorithm is responsible for converting structured meal data into accurate values for calories, proteins, carbohydrates, fats, fiber, and sodium.

**Input:** A matched meal \( M = \{(D_i, q_i, u_i)\}_{i=1}^{N} \), where each \( D_i \) is the matched nutrition database entry containing per-100g nutrient values.  
**Output:** Aggregated nutrient totals \( N_{meal} = \{kcal, protein, carbs, fat, fiber, sodium\} \) for the meal.

For each item, the algorithm first computes a quantity factor that depends on the unit. When the quantity is provided in grams, the factor is calculated as \( \alpha_i = q_i / 100 \), since the database stores nutrient values per 100 grams. When the quantity is given in serving units, the factor is taken directly as \( \alpha_i = q_i \), because the database also stores per-serving values for such items.

The nutrient contribution of each item is then computed as \( n_i = D_i \cdot \alpha_i \), which scales every macronutrient and supporting nutrient according to the actual quantity consumed. The final meal-level nutrient totals are obtained by aggregating across all items as \( N_{meal} = \sum_{i=1}^{N} n_i \). These totals are then persisted in the nutrient aggregates table for fast retrieval, contribute to the user’s daily totals, and form the foundation for the downstream daily summary and recommendation modules. This deterministic mathematical core ensures that the system produces consistent and reproducible nutritional values across all meals and users.

## 6.6 GPT Workflow

The GPT-driven workflow forms the intelligence backbone of the system and orchestrates how a user’s natural language input is transformed into accurate dietary insights. The flow begins when the user enters a message, typically describing a meal in conversational language. This input is first sent to the backend API, where it is preprocessed through cleaning, normalization, and basic validation to ensure that the message is structurally suitable for downstream processing. The cleaned text is then passed to the GPT-based Large Language Model, which uses its trained linguistic knowledge to interpret the meaning of the sentence and identify the user’s intent, such as logging a meal, asking for dietary advice, or requesting a daily summary.

Once the intent has been determined, the GPT model extracts important entities from the message, including food items, quantities, units, portion sizes, and preparation indicators. After this extraction, GPT produces a structured output, for example transforming a description like “two rotis with dal and a glass of milk” into a clean structured representation containing each item with its quantity. This structured output is then sent to the nutrition engine, which maps the extracted food items to entries in the curated food composition database and computes the corresponding calorie, protein, carbohydrate, fat, and supporting nutrient values.

After computation, the results are stored in the database under the user’s account, ensuring that every meal entry contributes to historical insights, daily aggregates, and long-term trends. The system can also forward the same data to the recommendation module, which compares actual intake with predefined goals and produces meaningful suggestions. Finally, a natural language response is generated and sent back to the user, for example confirming that a meal contributed 450 calories and providing a supportive dietary remark. This flow repeats for every user interaction, while maintaining contextual continuity so that subsequent conversations feel coherent, intelligent, and personalized.

## 6.7 Daily Summary Generation

The daily summary generation process is a specialized workflow that consolidates a user’s dietary activity into a structured, supportive, and actionable summary at the end of each day. The process begins by aggregating the total nutrient intake for the day across all logged meals, including calories, macronutrients, fiber, and other supporting components. These aggregated values are then compared against the user’s predefined nutritional goals to identify any surpluses, deficiencies, or imbalances that may have occurred during the day.

To enrich the summary with deeper context, the system also analyzes seven-day consumption trends, allowing it to recognize patterns such as repeated low-fiber intake, consistent high-sodium consumption, or steady alignment with calorie goals. Based on this combined daily and rolling analysis, the LLM generates a positively framed summary that highlights what went well, gently points out areas for improvement, and offers actionable suggestions that the user can realistically apply on the next day. The generated summary, along with relevant metadata such as the time of generation, comparison values, and feedback context, is then stored in the database for future reference, historical review, and recommendation refinement. This workflow transforms raw nutritional data into meaningful guidance that supports long-term healthy behavior change.

## 6.8 Retrieval-Augmented Generation (RAG) for Recommendations

In Phase 2 of the project, the system is designed to incorporate Retrieval-Augmented Generation in order to further improve the accuracy, factual grounding, and reliability of AI-generated recommendations. The RAG approach is built on the principle that a Large Language Model performs significantly better when its responses are grounded in trusted external knowledge rather than relying solely on its trained parameters. To enable this, a curated nutrition knowledge base is constructed and segmented into smaller passages, typically containing between 200 and 500 tokens, so that they remain semantically focused and easy to retrieve.

For each passage, vector embeddings are generated using a dedicated embedding model and stored in a vector database optimized for high-speed semantic search. When a recommendation needs to be generated, the system identifies the most relevant passages by performing similarity-based retrieval over the vector store, and these retrieved passages are then incorporated into the LLM prompt as supporting context. As a result, every recommendation produced through this pipeline is supported by credible nutritional knowledge rather than depending purely on the generative capabilities of the model. This design significantly reduces the risk of hallucination, increases trustworthiness, and makes the system better suited for sensitive health-related guidance.

## 6.9 Database Design

The system employs a normalized relational database schema designed to ensure strong data integrity, efficient querying, and long-term scalability. The users table stores all profile information, credentials, and personalized goals, and is linked through one-to-many relationships with the meals and recommendations tables, allowing every dietary record to be tied directly to its corresponding user. The meals table records each logged meal along with its timestamp and metadata, and connects to a meal items table that stores the individual foods consumed within each meal.

The meal items table references entries in the foods table, which contains a curated nutrition database of more than five hundred entries covering both general and culturally diverse food items. This structure makes it possible to retrieve detailed nutritional information for each meal item without redundant data storage. The nutrient aggregates table stores precomputed totals such as calories, proteins, carbohydrates, fats, fiber, and sodium for each meal, enabling rapid summary generation for daily and weekly views. The recommendations table stores AI-generated daily summaries and feedback associated with each user, completing the personalization loop within the database. Together, this schema ensures that the platform can support reliable storage, efficient retrieval, and clean expansion in future phases such as semantic search and advanced analytics.

## 6.10 API Design

The system exposes a structured set of RESTful API endpoints organized under the common base path `/api/v1/`, providing a clear, predictable, and version-controlled interface for both the frontend application and any future client integrations. The authentication endpoints handle critical security operations including user registration, login, and authenticated profile retrieval, ensuring that only verified users can access sensitive functionality. The user endpoints support profile updates and goal management, allowing users to evolve their personal information and dietary targets over time as their lifestyle changes.

The meal endpoints form the core of the system’s daily usage, supporting natural language meal logging, history retrieval with date filtering, individual meal access, and meal deletion when needed. The recommendation endpoints complete the intelligent guidance loop by providing daily summaries, on-demand recommendation generation, and feedback submission functionality. To enforce strict access control, all protected routes require JWT-based bearer token authentication, ensuring that every dietary interaction is tied to its rightful user and that personal data remains private throughout every API call. This API design provides a clean separation between authentication, user context, dietary data, and AI-driven outputs, making the system both maintainable and extensible.

## 6.12 Future Work: Image-Based Food Detection Algorithm (Phase 3)

To prepare the architecture for future multimodal capabilities, an image-based food detection algorithm has been planned as part of the Phase 3 enhancement roadmap. This algorithm is not part of the current MVP implementation but is included here to demonstrate forward-looking design. It would allow the system to detect food items directly from images captured by the user, eliminating the need for textual description in scenarios where typing is inconvenient. The algorithm receives an image frame \( I \) as input, either uploaded by the user or captured upon a sensor trigger, and produces structured output consisting of detected food objects with their class labels, bounding boxes, and confidence scores, which can then be forwarded into the existing nutrition pipeline.

The algorithm operates through a defined sequence of stages. In the preprocessing stage, the input image is resized to \( 640 \times 640 \) pixels and normalized as \( I_n = I/255 \) to ensure consistent model input. In the feature extraction stage, a YOLOv8 backbone is applied to the normalized image to generate multi-scale feature maps. The detection head then predicts bounding boxes \( B = \{(x_i, y_i, w_i, h_i)\}_{i=1}^{N} \) along with class probabilities \( P = \{p_{i,c}\} \) for each anchor.

The algorithm then performs confidence filtering using \( \text{Conf}_i = p_{obj,i} \cdot \max_c p_{i,c} \), discarding any detections below a defined threshold to remove unreliable predictions. Non-maximum suppression is applied by computing \( \text{IoU}(B_i, B_j) = \frac{B_i \cap B_j}{B_i \cup B_j} \) and suppressing detections where IoU exceeds \( \tau_{IoU} \). The final output is a consolidated set of detected food classes and their bounding boxes \( \{(c_i, B_i)\} \), which can be passed into the existing food matching and nutrient calculation algorithms. This design ensures that the future image-based logging feature integrates cleanly with the current text-based core without requiring architectural redesign.
